# Observability
## Observability signals/pillars (관측 데이터 유형/수단)
- Metrics (메트릭): 수치화된 지표 (CPU 사용률, 요청 수, 응답 시간 등)
- Logs (로깅): 이벤트 기록 (에러 메시지, 상태 변화, 사용자 액션 등)
- Tracing (트레이싱): 분산 시스템에서 요청 흐름 추적 (마이크로서비스 간 호출 관계, 지연 구간 파악 등)
- Profiling (프로파일링): 코드 실행 단위에서 성능 분석 (CPU/메모리 사용 패턴, 함수 호출 빈도 등)

# Monitoring
## 모니터링이란?
### 알림 alerting
문제가 발생한 시기나 시점을 파악하는 것이 모니터링에서는 가장 중요
문제가 생기면 모니터링 시스템은 책임 운영자를 호출할 수 있어야 함
### 디버깅 debugging
운영자는 문제의 근본 원인을 규명하고, 문제가 무엇이든지 간에 반드시 해결해야 함
### 추세 파악 trending
알림과 디버깅은 보통 몇 분부터 몇 시간까지 시간 단위로 발생
긴급하지 않은 문제인 경우에는 시스템이 어떻게 사용되고 시간에 따라 변화하는지를 확인할 수 있는 기능도 유용
추세 파악은 용량 계획 같은 설계 결정과 프로세스에 영향을 미칠 수 있음
### 플러밍 Plumbing
망치를 갖고 있으면, 모든 것이 못처럼 보이기 시작한다.
결국 모든 모니터링 시스템은 데이터 처리 파이프라인
때로는 맞춤형 솔루션을 만들기보단 모니터링 시스템의 일부를 다른 목적으로 적절하게 사용하는 편이 좋음
플러밍이 모니터링은 아니지만 실제로 흔한 경우이기 때문에 목록에 포함

## 모니터링 범주
### 프로파일링
프로파일링은 우리가 모든 시간에 대해 모든 이벤트의 컨텍스트를 가질 수 없지만, 제한된 기간의 일부 컨텍스트를 가질 수 있다는 방식으로 접근
데이터 양을 걱정해야 함
- 예시 tcpdump, 바이너리의 디버그 빌드, eBPF
### 트레이싱
트레이싱은 모든 이벤트를 살펴보는 것이 아니라, 관심 기능을 통과하는 일부 이벤트처럼 수백 개의 이벤트 중 특정 이벤트에만 집중
프로그램이 어느 부분에서 시간을 소비하고 어떤 코드 경로가 지연에 가장 큰 영향을 미치는지 알 수 있음
### 로깅
제한된 이벤트 집합을 살펴보고 각 이벤트에 대한 컨텍스트 일부를 기록
- 트랜잭션 로그
- 요청 로그
- 애플리케이션 로그
- 디버그 로그
### 메트릭
컨텍스트를 대부분 무시하고 다양한 유형의 이벤트에 대해 시간에 따른 집계를 추적, 항상 컨텍스트가 무시되는 것은 아니고 HTTP 요청의 경우엔 각 URL 경로마다 메트릭 설정 가능
프로세스당 1만 개의 메트릭 처리 정도가 합리적인 상한선
카디널리티 증가를 걱정해야 함
